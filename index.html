HTTP Headers & Version Negotiation

Every CDS endpoint carries its own independent version number, requested via the x-v header (mandatory) and optionally floored with x-min-v. The data holder must respond with the highest version it supports that is ≤ x-v and ≥ x-min-v, and echo the version actually served back in the response x-v header. If x-min-v is omitted or is ≥ x-v, it's ignored and only x-v applies.

ScenarioWhat happenedResult
x-v header absent entirelyMandatory header not sent400 — Header/Missing
x-v: foo or x-v: -1Present but not a positive integer400 — Header/InvalidVersion
x-v: 3 requested, but the endpoint only currently supports v5–v6Valid integer, but outside the range the holder currently implements (too old / retired, or too new / not yet released)406 — Header/UnsupportedVersion
x-v: 6, x-min-v: 4 requested, holder supports v5–v6Holder picks the highest version in range it supports200 — served as v6, response x-v: 6

Worked example — the classic "406 on GET Product Detail": Get Product Detail has gone through 7 versions since the standards launched (see the schedule below). A client hard-coded to x-v: 3 will 406 the moment v3 passes its retirement date, because the data holder is no longer required to serve it and v3 is outside the currently-supported range. The fix is never to hardcode a single version — send a sensible x-min-v floor (e.g. the oldest version your integration can tolerate) alongside the newest x-v you support, so the holder can negotiate down gracefully as versions are retired out from under you. Check the Endpoint Version Schedule below for the currently-active version of any endpoint.

Note 406 is also returned for a non-JSON/unsupported Accept header or an unsupported charset — that case isn't version-related and doesn't carry the Header/UnsupportedVersion code.

StatusMeaningNotes

304Not ModifiedStandard caching (ETag / If-Modified-Since)
